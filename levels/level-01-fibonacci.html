why does "0 stairs left" count as 1 way?

Picture the rabbit already standing on the top. That route is finished. How many successful routes did we just find? One.

0 stairs leftfinished route, count it as 1 success
1 stair leftonly one final hop works
2 stairs lefthop 1 + hop 1, or hop 2

So the base case is not a trick: 0 stairs left means "we made it."

what does this tree mean?

Each box is one little question: “how many ways with this many stairs left?” The arrows are the rabbit's first hop from that question.

5 stairs answer = ways(4) + ways(3)
4 stairs answer = ways(3) + ways(2)
3 stairs this appears again and again
0 stairs finished route, count +1

The red boxes are the same little question being asked again. That’s wasted work.

👀 see the pseudocode (step 1: brute force)
PSEUDO   brute force idea slow · repeats work 
idea: ask the same question for smaller stair counts

ways(stairsLeft):
    if stairsLeft is 0:
        return 1      # one route just finished

    if stairsLeft is 1:
        return 1      # only hop 1 works

    count_after_hop_1 = ways(stairsLeft - 1)
    count_after_hop_2 = ways(stairsLeft - 2)

    return count_after_hop_1 + count_after_hop_2

🎒 Backpack (answers we already found)

what does “tiny problem” mean?

The big problem is “5 stairs, how many ways?”

To answer it, we have to first know:

🪜 4 = ?a smaller version of the same puzzle
🪜 3 = ?an even smaller version
🪜 2 = ?tinier still
🪜 1 = 1easy! just one way
🪜 0 = 1already at the top, count this way

Now the idea appears naturally: if the same tiny question returns, do not think again. Look in the backpack.

👀 see the pseudocode (step 2: + memory)
PSEUDO   ask memory first ⬆ each tiny problem solved once 
ways(stairsLeft):
    if stairsLeft is 0 or 1:
        return 1

    if backpack has answer for stairsLeft:
        return backpack[stairsLeft]

    answer = ways(stairsLeft - 1) + ways(stairsLeft - 2)
    backpack[stairsLeft] = answer

    return answer

🧱 Tabulation: build a small table from scratch

Can we avoid the whole tree? Yes: start from the tiny obvious answers, then fill boxes from left to right. Each new box only asks: what were the last two answers?

Click Next to fill the next box →

step 0 / 0

what is “tabulation”?

It is just a scoreboard. Before filling a box, ask yourself:

Do I know 0 stairs?yes, already done = 1
Do I know 1 stair?yes, one hop = 1
How do I get 2?use the two boxes before it: 1 + 1

Then the rest of the row almost fills itself:

0 stairs → 1 1 stair → 1 2 stairs → 1+1 = 2 3 stairs → 1+2 = 3 4 stairs → 2+3 = 5 5 stairs → 3+5 = 8

We never recurse. We just walk the row left to right, adding the last two boxes. Done!

👀 see the pseudocode (step 3: no recursion at all)
PSEUDO   fill the scoreboard ⬆ no recursion · just a loop 
scoreboard[0] = 1    # finished path
scoreboard[1] = 1    # one final hop

for stair from 2 to top:
    scoreboard[stair] =
        scoreboard[stair - 1] + scoreboard[stair - 2]

return scoreboard[top]

what is the “space saver”?

Ask this at every step: what do I need to make the next answer?

prevthe answer two stairs back
currthe answer one stair back
nextprev + curr = the new answer

After that, the old prev is no longer needed. Slide forward: curr becomes prev, next becomes curr. That is the whole space saver.

👀 see the pseudocode (step 4: only 2 ints!)
PSEUDO   carry less memory ⬆ only 2 ints of memory 
prev = 1     # ways for 0 stairs
curr = 1     # ways for 1 stair

for each next stair:
    next = prev + curr
    prev = curr
    curr = next

return curr
